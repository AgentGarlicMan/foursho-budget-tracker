# Dark mode toggle and Philippine Peso formatting (v2)

This change adds a dark/light theme toggle (persisted to localStorage, respects system preference with a live listener) and replaces all dollar-formatted values with a shared `formatCurrency` utility producing `₱X,XXX.XX`. The Recharts chart elements are now explicitly themed via inline props conditioned on the current theme. All three issues from the v1 review have been addressed: chart theming is fixed, the system preference listener is in place, and the `toLocaleString('en-PH')` locale variance was acknowledged as acceptable for a browser-only app.

**Watch for:** Theme flash on first load — dark-mode users will see a white frame before React hydrates and the effect fires (likely). The system preference listener becomes permanently dead after the first manual toggle because `toggleTheme` always writes to localStorage (confirmed).

**Verdict**: APPROVED

## High-level view

The theme persistence logic has a subtle one-way door: once a user toggles the theme manually, localStorage is written, and the system preference listener's guard (`getStoredTheme() === null`) will never pass again. There's no "reset to system" action. This means the "follows system preference" behavior only applies to users who have never touched the toggle.

Dark-mode users who load the app for the first time will see a brief white flash. The `dark` class is set in a `useEffect` (runs after first paint). A blocking script in `index.html` that reads localStorage and sets the class before React renders would eliminate this.

## Theme flash for dark-preference users

The DOM class mutation happens in a `useEffect`:

```ts
useEffect(() => {
  const root = document.documentElement;
  if (theme === 'dark') {
    root.classList.add('dark');
  } else {
    root.classList.remove('dark');
  }
  // ...
}, [theme]);
```

React effects fire after the browser paints. On first load with a dark preference, the page renders with no `dark` class (white backgrounds), then the effect fires and the class is added, causing a visible flash. The standard fix for class-based dark mode in SPAs is a blocking `<script>` in `index.html`:

```html
<script>
  (function() {
    var stored = localStorage.getItem('capstone-budget-tracker-theme');
    var dark = stored === 'dark' || (!stored && window.matchMedia('(prefers-color-scheme: dark)').matches);
    if (dark) document.documentElement.classList.add('dark');
  })();
</script>
```

## System preference listener one-way door

The media query listener checks `getStoredTheme() === null` before applying the system change:

```ts
const handleChange = (e: MediaQueryListEvent) => {
  if (getStoredTheme() === null) {
    setTheme(e.matches ? 'dark' : 'light');
  }
};
```

But `toggleTheme` always persists to localStorage (via the `[theme]` effect), so after one toggle the stored key is permanently set and the listener becomes inert. This is a valid UX decision — explicit user choice overrides system preference — but there's no way back to "follow system" without clearing localStorage externally. Not blocking for the current scope, but worth noting if a "reset to system" feature is ever desired.
