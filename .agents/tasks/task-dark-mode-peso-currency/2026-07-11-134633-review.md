# Dark mode toggle and Philippine Peso formatting

This change adds two user-facing features to the budget tracker: a dark/light theme toggle that persists to localStorage and respects `prefers-color-scheme`, and a shared `formatCurrency` utility that replaces all inline dollar formatting with `â‚±X,XXX.XX`. The theme is implemented via a class-based dark mode strategy using Tailwind v4's `@custom-variant` directive, with the `dark` class toggled on `<html>`. Every component receives `dark:` utility variants for backgrounds, text, borders, and interactive states.

**Watch for:** The Recharts tooltip and chart grid/axes do not respond to dark mode, producing a white tooltip box and light-colored axis text against a dark background (confirmed). The `useTheme` hook does not listen for live system preference changes after initial load (confirmed). The `formatCurrency` function relies on `toLocaleString('en-PH')` which may produce inconsistent thousand-separator behavior across runtimes (possible).

## High-level view

The theme system uses `useTheme` to manage a `'dark' | 'light'` state, sync the `dark` class on `document.documentElement`, and persist to localStorage. The `@custom-variant dark` directive in `index.css` scopes `dark:` utilities to `.dark` ancestry. However, the hook doesn't react to live system preference changes once mounted â€” only the initial value is read from `matchMedia`.

Currency formatting is centralized in `formatCurrency`, which prepends `â‚±` and delegates number formatting to `toLocaleString('en-PH')`. All components consistently use this utility â€” no `$` symbols remain in the UI.

The Recharts chart component uses `formatCurrency` in its tooltip formatter, so tooltip *values* display correctly. But the chart's structural elements â€” axis labels, grid lines, tooltip container background â€” use Recharts defaults (white/light) with no dark mode awareness, producing a visually broken chart in dark mode.

## Recharts dark mode gap

The `ContributionChart` component applies `dark:bg-gray-800` to its container but the Recharts internals are unstyled:

```tsx
<CartesianGrid strokeDasharray="3 3" />
<XAxis dataKey="name" tick={{ fontSize: 12 }} />
<YAxis tick={{ fontSize: 12 }} />
<Tooltip formatter={(value: number) => [formatCurrency(value), 'Total']} />
```

`CartesianGrid` defaults to a light gray stroke. `XAxis`/`YAxis` ticks default to `#666` fill. The `Tooltip` renders a white background container with dark text. In dark mode, the grid becomes nearly invisible, axis labels are hard to read against `gray-800`, and the tooltip appears as a jarring white box.

The fix requires passing the current theme into the chart component and setting `stroke`, `tick={{ fill }}`, and `Tooltip contentStyle`/`labelStyle` accordingly. Since theme is already prop-drilled to Dashboard, it can be forwarded to `ContributionChart` without architectural changes.

## System preference listener missing

`useTheme` calls `getInitialTheme()` which checks `matchMedia('(prefers-color-scheme: dark)')` once. After that, theme state is purely driven by user toggle or stored value:

```ts
useEffect(() => {
  const root = document.documentElement;
  if (theme === 'dark') {
    root.classList.add('dark');
  } else {
    root.classList.remove('dark');
  }
  // ...
}, [theme]);
```

If a user has never toggled the theme (no localStorage entry) and changes their OS from light to dark, the app stays in light mode until a full page reload. A `matchMedia.addEventListener('change', ...)` listener that updates state when no stored preference exists would satisfy the "follow system preference" requirement more completely.

## `formatCurrency` locale dependency

```ts
export function formatCurrency(amount: number): string {
  return 'â‚±' + amount.toLocaleString('en-PH', {
    minimumFractionDigits: 2,
    maximumFractionDigits: 2,
  });
}
```

The `en-PH` locale isn't universally available. In environments where it's missing, `toLocaleString` falls back to the default locale, which could produce different grouping separators (e.g., periods instead of commas in some European locales). Since this is a client-side Vite app running in modern browsers, the risk is low for production, but it could surface in test environments or SSR scenarios. A defensive approach would be to use `Intl.NumberFormat` and check `resolvedOptions().locale` or simply hardcode the formatting for a single-currency app.
