# Firebase Realtime Database as shared backend

The budget tracker replaces its localStorage-only persistence with Firebase Realtime Database, keeping localStorage as a write-through cache and offline fallback. A `useFirebaseState` hook mirrors the signature of the existing `useLocalStorage` hook, so the rest of the app sees the same `{ state, setState, resetState }` interface with two additions: `loading` and `error`. A one-time migration pushes existing localStorage data to Firebase on first connection. Full-page `LoadingScreen` and `ErrorScreen` components gate the UI during initial fetch and on connection failure.

**Watch for:** The Firebase database is completely unauthenticated — anyone with the `databaseURL` can read and overwrite all app data (confirmed). The `setState` callback fires a Firebase write inside React's state updater function, which is a side-effect in a location React expects to be pure (confirmed). Write conflicts between tabs or users have no resolution strategy; last-write-wins with a stale-read guard that can itself miss updates (likely).

## High-level view

The hook replaces `useLocalStorage` drop-in style, maintaining the same return shape so Dashboard and its children are unchanged. Firebase is a single live `onValue` listener on `/appState`, with a 10-second timeout that triggers fallback to localStorage. Writes go to both Firebase and localStorage simultaneously.

The migration path is one-directional: localStorage to Firebase, triggered once when the Firebase path is empty. If multiple clients with divergent localStorage connect concurrently before any has migrated, whichever writes first wins — the other's data is silently lost.

There is no authentication layer, no database security rules, and no per-user data isolation. The entire app state lives at a single flat path readable and writable by anyone who obtains the project config.

The `setState` implementation dispatches a Firebase `set()` call from inside a React state updater function. React state updaters are expected to be pure; the side-effect works today but violates the contract and will double-fire under strict mode.

<summary>Issues (5)</summary>

1. **No authentication or database rules** — Firebase is open to the internet. Add Firebase Auth (even anonymous auth) and configure Realtime Database security rules to restrict reads/writes. At minimum, document that this is intentionally open during development.
2. **Side-effect inside state updater** — `setState` calls `set(dbRef, newState)` inside the `setLocalState` updater. Move the Firebase write outside the updater, or use a `useEffect` that reacts to state changes, to avoid issues under concurrent/strict mode.
3. **Write-flag race on concurrent updates** — `isWriting` is a single boolean ref. Rapid successive writes from the same client can clear the flag (via `.finally()`) before the earlier write's echo arrives back from `onValue`, causing the listener to incorrectly apply stale server state.
4. **Migration race between clients** — Two tabs with different localStorage data can both see Firebase as empty and both attempt migration. Only one write succeeds, the other's data is silently lost. Consider using a Firebase transaction for the migration write.
5. **ErrorScreen blocks interaction permanently** — When `error` is set, App renders only `ErrorScreen` with a reload button. The hook already falls back to localStorage data, but the UI never shows it. The user must manually reload even though valid cached data is already in state.

## No auth, no rules, no isolation

No security rules file is shipped with the project, and `initializeApp` is called without any auth provider. The real behavior depends entirely on whatever rules the Firebase project has configured server-side. If those rules are set to `".read": true, ".write": true` (common during prototyping), any HTTP client can GET or PUT the entire application state at `/appState`.

The absence of any mention of security rules in the codebase or documentation means a deployer has no guidance on what to configure. Even a comment in `firebase.ts` or a section in `.env.example` pointing to the required rules would close this gap.

## Side-effect inside React state updater

In `setState`:

```typescript
setLocalState((prev) => {
  const newState = typeof action === 'function' ? action(prev) : action;
  isWriting.current = true;
  const dbRef = ref(database, '/appState');
  set(dbRef, newState)           // <-- side-effect
    .catch(...)
    .finally(...);
  saveToLocalStorage(newState);  // <-- side-effect
  return newState;
});
```

In strict mode (enabled by default in development with React 19), React intentionally double-invokes state updaters to surface impurities. This means `set(dbRef, newState)` fires twice for each logical state change during development, and the `isWriting` flag will be set/cleared unpredictably.

A cleaner pattern: compute `newState` inside the updater, return it, and fire the Firebase write in a `useEffect` watching `state`, or call `setLocalState` then fire the write sequentially outside the updater.

## Write-flag race window

The `isWriting` ref prevents the `onValue` listener from applying Firebase echoes of the client's own writes, but the flag is a single boolean with no correlation to specific writes:

1. User triggers `setState` → `isWriting = true`, Firebase `set()` dispatched.
2. User immediately triggers another `setState` → `isWriting = true` again (same value).
3. First write completes → `.finally()` sets `isWriting = false`.
4. Firebase echoes the first write via `onValue` → listener sees `isWriting === false`, applies it, potentially reverting the second write's optimistic local state.

For a budget tracker with low write frequency this window is narrow, but rapid additions (clicking quickly) can cause state to flicker or revert entries.

## Migration race between concurrent clients

```typescript
if (isEmptyState(data) && !hasMigrated()) {
  const localData = getLocalStorageState();
  if (localData && !isEmptyState(localData)) {
    set(dbRef, localData);
  }
}
```

Two browser tabs (or two team members) opening the app for the first time will both see `isEmptyState(data) === true` and `hasMigrated() === false`. Both call `set(dbRef, localData)` with potentially different data. Firebase `set()` is last-write-wins — no transaction, no compare-and-swap. Both clients' `.then()` fires successfully, so both mark themselves as migrated despite only one's data surviving.

## ErrorScreen blocks access to cached data

When the `onValue` error callback fires, the hook sets `error` and loads localStorage fallback into `state`. But `App.tsx` renders:

```tsx
if (error) {
  return <ErrorScreen error={error} />;
}
```

The `state` variable already contains valid localStorage data, but it's never rendered. The user sees "Connection Error" with a reload button. A non-blocking error banner showing the cached data would allow offline use without a full page reload.
