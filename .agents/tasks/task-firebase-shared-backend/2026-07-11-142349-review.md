# Firebase Realtime Database as shared backend (v2)

The budget tracker replaces localStorage-only persistence with Firebase Realtime Database, retaining localStorage as a write-through cache and offline fallback. A `useFirebaseState` hook mirrors the existing `useLocalStorage` API, and the rest of the app is unchanged. This v2 pass addresses all five issues from the v1 review: security documentation added, Firebase writes moved to a microtask outside the state updater, a `pendingWrites` counter replaces the single boolean flag, migration uses `runTransaction` for race safety, and a non-blocking `ErrorBanner` replaces the full-page block when cached data exists.

**Watch for:** The `pendingWrites` counter fixes the single-boolean race but introduces a window where *all* server echoes are suppressed while any write is in-flight, meaning a concurrent update from another client is silently dropped until the local write settles (confirmed). The `hasData` check in App.tsx uses array length, so a user who has cleared all entries sees the blocking `ErrorScreen` even though the app is functional with empty state (confirmed). The `saveToLocalStorage` call remains synchronous inside the state updater, a minor purity violation under strict mode (possible).

**Verdict**: APPROVED

## High-level view

The hook uses a `pendingWrites` counter that increments before each write and decrements in `.finally()`. While any write is pending, the `onValue` listener returns early, preventing echo-induced flicker. The trade-off: legitimate remote updates arriving during that window are also discarded. For a low-frequency budget tracker this is acceptable; for a high-write-rate collaborative app it would not be.

The `setState` callback defers the Firebase `set()` via `Promise.resolve().then()`, keeping the async network call out of the React updater. The synchronous `saveToLocalStorage` call remains inside the updater — under strict mode double-invocation it writes the same value twice, which is harmless but technically impure.

Migration uses `runTransaction` with an `isEmptyState` guard: the transaction only commits if the server path is still empty, so concurrent tabs no longer race. `markMigrated()` fires regardless of commit outcome, preventing repeated attempts.

The error UX gates on a `hasData` check (`contributions.length > 0 || expenses.length > 0`). When cached data exists, the app renders with a dismissible `ErrorBanner`. The gap: empty-state users (fresh start or post-reset) hit the blocking `ErrorScreen` on any connection hiccup, even though an empty dashboard is perfectly usable.

## PendingWrites counter and echo suppression

```typescript
if (pendingWrites.current > 0) {
  return; // ignore onValue echo
}
```

While *any* local write is in-flight, a remote update from another tab or user that arrives via `onValue` is silently ignored. Once all local writes settle and `pendingWrites` hits zero, the next `onValue` fires with whatever Firebase currently holds — so the remote update is eventually picked up, but there's a staleness window proportional to the Firebase round-trip time. In a multi-user scenario with near-simultaneous writes, one user's update can be invisible to the other for the duration of a full write round-trip.

## setState microtask pattern

```typescript
setLocalState((prev) => {
  const newState = typeof action === 'function' ? action(prev) : action;
  saveToLocalStorage(newState);       // sync, inside updater
  pendingWrites.current++;
  Promise.resolve().then(() => {      // async, outside updater
    set(dbRef, newState)...
  });
  return newState;
});
```

The `pendingWrites` increment happens synchronously inside the updater — correct, because the counter must be high before the next `onValue` callback fires. The `saveToLocalStorage` call is also synchronous inside the updater; under strict mode double-invocation this means two identical localStorage writes per logical state change. No corruption results, but a `useEffect`-based approach would eliminate both impurities if strict-mode compliance becomes a goal.

## ErrorBanner and hasData gating

```typescript
const hasData = state.contributions.length > 0 || state.expenses.length > 0;

if (error && !hasData) {
  return <ErrorScreen error={error} />;
}
```

A brand-new user (or one who just reset data) has both arrays empty, so `hasData` is false. If Firebase times out on their next visit, they see the blocking ErrorScreen even though the app would render a usable empty dashboard. A more robust signal: a `hasSynced` boolean set to true after the first successful `onValue` response, or simply always using the non-blocking ErrorBanner.
